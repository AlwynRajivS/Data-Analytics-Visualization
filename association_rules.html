📖 Fundamentals
Association Rule Mining
Association Rule Mining (ARM) discovers interesting relations between variables in large datasets. It identifies frequent if-then patterns and quantifies their strength using support, confidence, and lift — enabling powerful data-driven decisions.
Core Concepts
📦
Itemset
A collection of one or more items. e.g. {HighMarks, HighAttendance} — any subset of all possible items in the dataset.
📏
Frequent Itemset
An itemset whose support ≥ minimum support threshold. Only frequent itemsets are used to generate valid association rules.
➡️
Association Rule
An implication X → Y where X (antecedent) and Y (consequent) are disjoint itemsets. "If X, then Y."
🔍
Strong Rule
A rule satisfying both minimum support AND minimum confidence thresholds simultaneously.
🛒
Transaction
A single record in the database — e.g. one customer purchase, one student's profile — containing a set of items.
🗄️
Transaction DB
The full collection of all transactions. Denoted as T with N transactions total, used to compute all supports.
Key Metrics
Support(X→Y) = count(X∪Y) / N
Confidence(X→Y) = count(X∪Y) / count(X)
Lift(X→Y) = Confidence(X→Y) / Support(Y)
Conviction = (1 − Support(Y)) / (1 − Confidence(X→Y))
Leverage(X→Y) = Support(X∪Y) − Support(X) · Support(Y)
Lift > 1
Positive correlation — X and Y appear together more often than by chance. Rule is useful.
Lift = 1
X and Y are statistically independent — the rule has no predictive power.
Lift < 1
Negative correlation — X and Y appear together less than expected by chance.
Types of Association Rules
Single-Dimensional
  • Rules involve only one attribute dimension
  • Example: buys(bread) → buys(butter)
  • Simple and fast to mine from market data
  • Limited expressiveness for rich datasets
Multi-Dimensional
  • Rules span two or more attributes at once
  • Example: age=young ∧ dept=CS → marks=high
  • Richer, more actionable cross-attribute patterns
  • Requires discretization of numeric attributes
Multilevel
  • Rules mined at different abstraction levels
  • General rules: high support, less specific
  • Specific rules: low support, very detailed
  • Uses concept hierarchies to navigate levels
Real-World Applications
🛒
Market Basket
Discover product co-purchase patterns to optimize shelf placement and cross-sell promotions.
🏥
Healthcare
Find co-occurring symptoms or diagnoses to aid clinical decision-making and treatment plans.
🎓

Identify student behavior patterns — e.g. high attendance + lab work → high grades.
🌐

⚙️ Algorithm
The Apriori Algorithm
Apriori (1994, Agrawal & Srikant) mines frequent itemsets using the anti-monotonicity principle: any subset of a frequent itemset must also be frequent — enabling massive pruning of the candidate search space.
The Apriori Property
Anti-Monotonicity Principle
If itemset {A, B} is infrequent, then every superset containing it — {A, B, C}, {A, B, D} — is also infrequent and can be pruned without scanning the database. This dramatically reduces the search space at each level.
Step-by-Step Process
1
Scan DB → Find Frequent 1-Itemsets (L₁)
Scan the entire database once. Count occurrences of each individual item. Remove items whose count falls below min_support × N.
min_sup=40%, N=10 → min_count=4
Item A:8✓ B:6✓ C:3✗ D:5✓ → L₁={A,B,D}
2
Generate Candidates C₂ from L₁
Join L₁ with itself to generate all 2-item candidate pairs. Then prune any candidate whose subset is not in L₁.
C₂ = {A,B}, {A,D}, {B,D} → scan DB → count each
3
Prune → Get L₂
Compare each candidate count against min_count. Remove those below threshold — these are infrequent 2-itemsets.
{A,B}:5✓ {A,D}:4✓ {B,D}:2✗ → L₂={{A,B},{A,D}}
4
Generate C₃ from L₂ and Repeat
Continue generating Cₖ candidates, scanning the DB, filtering to Lₖ. Repeat until Lₖ is empty.
C₃={A,B,D} → subset {B,D}∉L₂ → PRUNED → L₃=∅ → STOP
5
Generate Association Rules
For each frequent itemset I where |I|≥2, generate all non-empty subsets X. Create rule X→(I−X). Keep rule if confidence ≥ min_conf.
From {A,B}: A→B conf=5/8=62.5%✓ | B→A conf=5/6=83%✓
From {A,D}: A→D conf=4/8=50% | D→A conf=4/5=80%✓

Limitations & Alternatives
Apriori Weaknesses
  • Requires multiple full database scans
  • Generates huge numbers of candidates
  • Memory-intensive on dense datasets
  • Slow on very large transaction databases
FP-Growth (Better)
  • Only 2 database scans needed
  • Uses a compressed FP-tree structure
  • No explicit candidate generation
  • Much faster on large sparse datasets
ECLAT Algorithm
  • Uses vertical data format (TID-lists)
  • Intersection-based counting
  • Better memory locality on some data
  • Good for moderate-size databases
🌳 Concept
Multilevel Association Mining
Mining at various levels of a concept hierarchy. Higher levels yield general rules with stronger support; lower levels yield specific rules with richer context and actionable detail.
What is a Concept Hierarchy?
🏫
Level 0 — Root
The most general level: all records together. e.g. "All Students" or "All Products."
📂
Level 1 — Category
Broad groupings: Department (CS, ECE, MECH) or Product Category (Electronics, Food).
🏷️
Level 2 — Subcategory
Finer breakdowns: CS → {Male, Female} or Electronics → {Phones, Laptops}.
🔬
Level 3 — Leaf
Most specific: CS ∧ Female ∧ Hostel → individual record groups or specific products.
Concept Hierarchy Examples
Education Dataset Hierarchy
Level 0: All Students
Level 1 (Dept): CS · ECE · MECH
Level 2 (Gender): CS ∧ Male · CS ∧ Female
Level 3 (Residence): CS ∧ Female ∧ Hostel

L2 rule: CS ∧ Female → HighMarks (sup=28%, conf=84%)
L3 rule: CS ∧ Female ∧ Hostel → HighMarks (sup=14%, conf=91%)

Level 0: All Transactions
Level 1 (Category): Food · Electronics · Clothing
Level 2 (Subcategory): Food ∧ Beverage · Food ∧ Dairy
Level 3 (Brand): Food ∧ Dairy ∧ Amul

L1: Food → Beverage (sup=55%) — very general
L3: Dairy ∧ Bread → Butter (sup=12%) — highly specific
Support Threshold Strategies
Uniform Support
  • Same threshold across all levels
  • Simple and easy to implement
  • May miss specific rare rules at deep levels
  • Works well when data is evenly distributed
Reduced Support
  • Lower threshold at deeper hierarchy levels
  • Captures rare but meaningful patterns
  • Compensates for sparser data at leaf nodes
  • Best for most real-world hierarchical mining
Group-Based Support
  • Different thresholds per category group
  • Most flexible and adaptive approach
  • Requires domain knowledge to set thresholds
  • Used in advanced commercial systems
Cross-Level Rule Mining
Mining Across Levels Simultaneously
Rules can span multiple hierarchy levels simultaneously:

CS (L1) ∧ Hostel (L2) → HighMarks (L0)

This "cross-level" approach is more expressive but computationally costlier. The support threshold is typically set to the minimum among the levels involved.
🔷 Concept
Multidimensional Association Mining
Rules spanning two or more attributes simultaneously — revealing cross-attribute patterns invisible in single-column analysis. Enables discovery of complex conditional relationships.
Types of Multidimensional Rules
↔️
Inter-Dimension
Each attribute appears exactly once in the rule. Example:
age=young ∧ dept=CS → marks=high
🔁
Hybrid Rules
Some attributes appear multiple times at different levels within the same rule, crossing hierarchies.
📐
Quantitative
Involves discretized numeric attributes. Values binned into ranges: marks=[80-100] or age=[18-22].
🏷️
Categorical
Nominal attributes with no inherent order: gender=Female, dept=CS, city=Chennai.
Cross-Tabulation (Heatmap) Analysis
Reading the Cross-Tabulation Matrix
Cell(row, col) = count of records where both dimension values co-occur simultaneously.

Darker cells = more records = stronger co-occurrence frequency.
Hot spots (high values) reveal ideal rule candidates — pairs worth mining further.

Example: Cell(CS, HighMarks)=45 out of 100 CS students → 45% support for CS∧HighMarks pattern.
Handling Quantitative Attributes
Equal-Width Binning
  • Divide range into N equal-width intervals
  • Easy to implement, interpretable
  • Can misrepresent skewed distributions
Equal-Depth Binning
  • Each bin contains equal number of records
  • Better for skewed data distributions
  • Used in quantitative ARM systems
Clustering-Based
  • Natural groupings from the data itself
  • Most statistically sound approach
  • Computationally expensive to compute

Complete Metrics Reference
All metrics used in association rule mining with precise formulas, value ranges, and interpretation guidelines for choosing the right threshold for your dataset.
Formulas & Ranges
Support(X→Y) = |X∪Y| / N range: [0, 1]
Confidence(X→Y) = |X∪Y| / |X| range: [0, 1]
Lift(X→Y) = Confidence / Support(Y) range: [0, ∞)
Leverage(X→Y) = Supp(X∪Y) − Supp(X)·Supp(Y) range: [−1, 1]
Conviction(X→Y) = (1−Supp(Y)) / (1−Conf) range: [0, ∞)
Jaccard(X,Y) = Supp(X∪Y) / (Supp(X)+Supp(Y)−Supp(X∪Y)) range: [0, 1]

Choosing Thresholds
High Support (>50%)
Very common patterns. Few rules generated. Extremely high quality — only the most dominant associations survive.
Medium Support (20–50%)
Balanced discovery. Best starting point for most datasets. Good variety of rules at reasonable quality.
Low Support (<20%)
Rare pattern discovery. Combine with high confidence (>80%) to avoid noise. Used for niche segments.
High Confidence (>80%)
Only very reliable rules survive. Best for prediction tasks. Fewer rules, higher actionability.
Common Pitfalls
Simpson's Paradox
  • A trend appears in groups but reverses overall
  • Always check subgroup rules separately
  • Combine with stratified analysis
Too Many Rules
  • Lower support → exponential rule explosion
  • Use post-processing: rank by lift or leverage
  • Apply closed/maximal itemset constraints
Spurious Rules
  • High confidence but low lift = misleading
  • Always check lift > 1 for true association
  • Use conviction for directional validation
